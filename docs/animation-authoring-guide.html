Player Animation — Blender Authoring Guide

Personal reference for hand-authoring the 13 clips that drive the player AnimationTree.
Companion to docs/superpowers/specs/2026-05-24-animation-state-machine-design.md (the formal spec).

Intro #

The state-machine spec defines what clips are needed and how they wire into the AnimationTree. This guide defines how to author each clip in Blender so it works correctly at runtime — length, loop discipline, pose blocking, sync alignment, additive considerations.

Read alongside docs/asset-pipeline.md (export discipline) and docs/blender-mcp-guide.md (if using MCP-assisted authoring). This guide assumes Blender 4.x+ familiarity — it doesn't cover Blender basics, only project-specific requirements.

Total: 13 clips. All authored as Blender Actions on a single rigged character, exported in one .glb file.

Rig assumptions #

The character is bipedal humanoid. Rig requirements:

  • Standard bone hierarchy: root → hips → spine → chest → neck → head, plus arms (shoulder/upper-arm/forearm/hand) and legs (thigh/shin/foot/toe). Rigify or hand-built; either works.
  • Bone naming: glTF-compatible. Avoid Blender's .L/.R if it causes issues downstream; _l/_r suffixes are safer cross-tool.
  • Bind pose: T-pose or A-pose, facing +Y in Blender (which becomes -Z in Godot after the glTF Y-up axis swap). This matches Godot's canonical -Z forward convention. (Earlier versions of this guide said +Z forward — the project flipped to canonical -Z on 2026-05-25; see docs/godot-gotchas.md.)
  • Root bone at origin in bind pose. Hips above root, never below it.
  • Scale: apply all scale transforms before rigging (Object → Apply → All Transforms). Character height ≈ 1.7m in Blender units = 1.7m in Godot.
No root motion. All character movement is driven by the physics controller. Animations are in-place gait cycles — never key the root bone position to translate forward. Keying root rotation/Y-translation for breathing or bounce is fine; horizontal translation is not.

Export checklist #

Per docs/asset-pipeline.md. Key points:

  • File: assets/models/player.glb (binary glTF 2.0).
  • Animations: all 13 clips exported as named Actions. Use Blender's NLA editor or "Push Down" each Action into its own strip — confirm each Action is pushed before export.
  • Clip names MUST match the spec exactly. These names become the AnimationPlayer animation names in Godot, which the AnimationTree's BlendSpace and StateMachine nodes reference by string. A typo silently breaks the binding.
  • Frame rate: Blender scene at 30 fps. Confirm in Render → Frame Rate before keying.
  • glTF export settings: Include Selected Objects = off (export the whole rig); Animation → Always Sample Animations = on; Animation → Group by NLA Track = on (preserves your action names).

Authoring discipline #

Frame rate

30 fps across all clips. Don't mix rates between actions.

Looping clips

For any clip marked "Loop: yes": first frame == last frame. The easiest path is to copy frame 0's keyframes to frame N (where N = your intended cycle length + 1), then in the AnimationPlayer-side import, set the clip's end to frame N so the last frame is consumed only for loop wraparound.

For BlendSpace anchors, looping discipline is critical — feet skate on the blend if walk and run cycles desync.

Sync groups

The AnimationTree skill notes: never use Sync groups for animations with wildly different lengths. Two BlendSpaces in this design use sync alignment:

  • Locomotion BlendSpace2D (Move state): all 7 anchors (idle, walk_fwd, walk_lean_L/R, run_fwd, run_lean_L/R) blend together. Author so:
    • idle can be any length — it's the center anchor, doesn't conflict with gait phase.
    • walk_fwd / walk_lean_L / walk_lean_R all have identical cycle length (e.g., 30 frames) and aligned foot strikes (left heel down at frame 0, right heel down at frame 15).
    • run_fwd / run_lean_L / run_lean_R all have identical cycle length (e.g., 18 frames) and aligned foot strikes (left at 0, right at 9).
    • Walk and run cycles can differ in length — the AnimationTree sync group time-scales them to a common phase. Just keep walk-variants matched and run-variants matched within their tier.
  • Drift BlendSpace1D: drift_walk and drift_run should be loosely sync-aligned (foot pattern roughly in phase), but the drift state is short-duration so exact alignment matters less.

Lean offsets

The 4 lean variants (walk_lean_L/R, run_lean_L/R) are derivatives of their _fwd counterparts, not from-scratch animations. Two production paths:

  1. Copy + tweak: duplicate the Action, add an additive bone-rotation layer on torso/spine/head/hips to introduce the lean. Keep foot strikes identical.
  2. Additive in Blender: author a "lean offset" Action separately (just torso rotation, no leg motion), then use NLA strip blend mode = Add over walk_fwd at export. Result is two Actions per lean variant — more work but cleaner.

Path 1 is simpler and ships faster. Path 2 is more maintainable if you want to retune the lean later.

The additive clip (dash_burst)

One clip in this set is additive at runtime via the AnimationTree's OneShot Add node: dash_burst. Authoring rules differ:

  • The clip's frame 0 should be the rig's bind pose (or very close to it). The OneShot Add node interprets the clip as deltas relative to the locomotion underneath.
  • Don't animate legs in dash_burst — the locomotion legs are still cycling underneath. Restrict the action to torso, arms, head, hair/cape.
  • Preview the additive blend in Blender by stacking the dash_burst NLA strip on top of walk_fwd or run_fwd with blend mode = Add. If it looks weird in Blender, it'll look weird in Godot.

Per-clip authoring sheets #

idle

Length
~75 frames (2.5s @ 30fps), flexible
Loop
Yes — first frame == last frame
Root motion
None (in-place)
Sync siblings
None (anchor in Move BlendSpace2D + Drift BlendSpace1D both reference this same clip)
Pose blocking
Standing tall, weight even on both feet. Slow breath cycle: chest rises slightly on inhale, settles on exhale. Optional micro idle motion in arms/head — keep subtle.

Notes: This is the most-visible clip — it plays at startup, after every stop, after every hit recovery. Don't make it fidgety; over-animated idle reads as "twitchy" when sustained. Used by both the Move and Drift BlendSpaces, so it has to read as the "settled" pose for both contexts.

walk_fwd

Length
30 frames (1.0s @ 30fps) — one full gait cycle
Loop
Yes — first frame == last frame
Root motion
None (in-place)
Sync siblings
walk_lean_L, walk_lean_R (must be identical length, aligned foot strikes)
Pose blocking
Relaxed forward walk. Frame 0: left heel strike, right foot pushing off. Frame 15: right heel strike, left foot pushing off. Frame 29 → 0: cycle wraps. Arms swing opposite to legs (left arm forward when right leg forward).

Notes: Foundational clip for all locomotion. Sets the tempo for the entire walk tier. Foot strikes at 0 and 15 are the anchor phase — lean variants must hit the same frames.

walk_lean_L

Length
30 frames (identical to walk_fwd)
Loop
Yes
Root motion
None
Sync siblings
walk_fwd, walk_lean_R (matched length, aligned foot strikes at 0 and 15)
Pose blocking
walk_fwd with torso leaned ~15–20° to the character's left. Shoulder line tilts, hips counter-shift slightly right for balance. Head can follow the lean or stay level (depends on style). Arm swing weighted slightly leftward.

Notes: Derive from walk_fwd via duplicate-and-overlay. The lean must read at top-down camera angles — shoulder tilt is the dominant cue. Don't add yaw rotation to the spine; the controller handles all yaw.

walk_lean_R

Length
30 frames (identical to walk_fwd)
Loop
Yes
Root motion
None
Sync siblings
walk_fwd, walk_lean_L
Pose blocking
Mirror of walk_lean_L: torso leaned right, hips shift left, etc.

Notes: If the rig is symmetric, Blender's "Paste Pose Flipped" can produce this from walk_lean_L's keyframes.

run_fwd

Length
18 frames (0.6s @ 30fps) — one full gait cycle
Loop
Yes
Root motion
None (in-place)
Sync siblings
run_lean_L, run_lean_R (identical length, foot strikes at 0 and 9)
Pose blocking
Forward run. Longer stride than walk, more forward lean of torso, larger arm swing, brief airborne phases between foot strikes. Frame 0: left heel strike, right foot in airborne push-off. Frame 9: right heel strike, left in push-off.

Notes: Must blend cleanly with walk_fwd. The AnimationTree sync group time-scales between walk's 30-frame cycle and run's 18-frame cycle as speed transitions. Foot phase alignment (left at 0, right at midpoint) keeps the blend coherent — don't put left strike at frame 0 in walk but frame 9 in run.

run_lean_L

Length
18 frames (identical to run_fwd)
Loop
Yes
Root motion
None
Sync siblings
run_fwd, run_lean_R
Pose blocking
run_fwd with torso leaned ~20–25° left. Lean angle can be stronger than walk variants because the running pose already has more forward lean — the additive lean compounds. Arm swing favors the lean side.

Notes: Same derivation strategy as walk_lean_L.

run_lean_R

Length
18 frames (identical to run_fwd)
Loop
Yes
Root motion
None
Sync siblings
run_fwd, run_lean_L
Pose blocking
Mirror of run_lean_L.

drift_walk

Length
~30 frames (1.0s @ 30fps), flexible
Loop
Yes — but loop point less critical (drift is typically short-duration before transitioning to idle)
Root motion
None
Sync siblings
drift_run (Drift BlendSpace1D group)
Pose blocking
"Coasting" / "post-stop" body language. Upper body leaned slightly backward (opposite of forward locomotion). Arms trail behind or hang loose. Knees slightly bent for braking. Foot pattern can be a slow drag, a relaxed step-and-hold, or a glide pose with minimal foot motion.

Notes: No library template for this — "coast after releasing input" isn't a standard mocap clip. Most likely path is to author from drift-specific reference (ice skater gliding to stop, character sliding on loose surface). This is the clip that gives drift its distinct visual identity vs. walk.

drift_run

Length
~18 frames (0.6s @ 30fps), flexible
Loop
Yes (same loop-tolerance note as drift_walk)
Root motion
None
Sync siblings
drift_walk
Pose blocking
drift_walk's coast pose at higher speed. More aggressive backward lean. Faster but lazier foot pattern — heels lead, character is "letting momentum carry them."

Notes: Same authoring approach as drift_walk. The lean-back amount should feel proportional to the speed it's coasting from.

pivot_skid

Length
~10–12 frames (0.33–0.4s @ 30fps)
Loop
No — one-shot, auto-advances to Move at end
Root motion
None
Sync siblings
None directly, but the AnimationTree uses Sync switch mode entering Pivot — so the clip's start frame should make sense layered over the running gait's foot-plant phase
Pose blocking
Front-loaded: first 1/3 (frames 0–3) is the skid impact — heel jams down, body rotates AWAY from the new direction (anticipation pose), arms windmill or brace. Middle 1/3 (frames 3–7): body re-orients toward the new direction, foot pivots. Final 1/3 (frames 7–11): settle into neutral, ready to transition to Move.

Notes: The most expressive single clip in the set. Anticipation pose (body twisting opposite the new direction) is critical for making the pivot feel impactful — without it, the skid reads as a teleport. Reference: NBA player pivoting on a hard cut, or Mario's skid animation.

dash_burst additive

Length
~10 frames (0.33s @ 30fps) — matches controller's impulse_decay tail
Loop
No — one-shot, OneShot Add at BlendTree root
Root motion
None
Sync siblings
None (overlay clip; runs in parallel to whatever locomotion state is active)
Pose blocking
Upper body / arms / torso only. Frame 0: rig bind pose (zero delta). Frames 1–3: brace pose — torso braces forward into the dash, arms pull back, head dips. Frames 4–6: hold or slight follow-through. Frames 7–10: ease back to bind pose. Do not animate legs — they're cycling in the locomotion state underneath.
Additive authoring is the trickiest in the set. Frame 0 must equal the bind pose, otherwise the OneShot Add will offset the locomotion result by the difference. Preview in Blender by stacking the dash_burst NLA strip with blend mode = Add over walk_fwd and run_fwd — if it doesn't look right there, it won't look right in Godot.

Notes: If you also want a hair/cape flare, animate the cloth bones here — they're not part of locomotion so additive layering won't conflict.

hit_flinch stub

Length
~12 frames (0.4s @ 30fps)
Loop
No — one-shot, auto-advances to Idle at end
Root motion
None
Sync siblings
None
Pose blocking
Short flinch. Frames 0–3: torso recoils slightly, head jerks back, arms come up defensively (forearm cross or guard pose). Frames 4–8: hold. Frames 9–11: relax back toward neutral.

Notes: Authoring quality bar is intentionally low — this is a placeholder until the damage system spec exists and you decide whether to do directional variants (hit_front / hit_back / hit_L / hit_R). Don't over-invest now.

Authoring order #

Recommended order — earlier clips inform later ones, dependencies first.

  1. idle — sets the rest pose, body shape, personality. Everything else flows from this baseline.
  2. walk_fwd — establishes the walk gait baseline. Get foot timing and stride feel right here; all walk variants inherit.
  3. run_fwd — extends the gait at speed. Confirm walk and run blend cleanly by previewing a manual interpolation between them in Blender's NLA.
  4. drift_walk + drift_run — derive from walk_fwd / run_fwd with upper-body coast adjustment. Tackle them as a pair so the lean and pacing feel consistent.
  5. walk_lean_L, walk_lean_R, run_lean_L, run_lean_R — derive from forward variants with lean offsets. Do L then mirror to R via Paste Pose Flipped.
  6. pivot_skid — high-effort single clip. Worth iterating on; this is the most-noticed pose-only moment in the locomotion loop.
  7. hit_flinch — quick stub. 30 minutes max; replace later.
  8. dash_burst — last. Requires understanding how additive layering reads. Test in Blender's NLA against walk_fwd and run_fwd before committing.

After the full set is in Blender and previewing cleanly, export to assets/models/player.glb, then move to the Godot AnimationTree wiring step (see the spec's "Implementation order" section).
